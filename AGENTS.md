# AGENTS.md - правила написания Helm-чартов для ИИ-агентов

Это операционная инструкция для ИИ-агента, который заводит или правит Helm-чарт в
репозитории `console-charts`. Читается сверху вниз: сначала правила, которые нельзя
нарушать, затем пошаговая процедура, затем пофайловый разбор и типичные ошибки.

Документ не заменяет два других источника, а связывает их в порядок действий:

- `CLAUDE.md` - правила процесса репозитория (язык коммитов, git-workflow, табу).
- `CONVENTIONS.md` - подробный справочник по устройству чарта (нейминг ресурсов,
  реестр `kindShort`, схема, хелперы, файлы значений). Это reference: когда нужен
  «как именно» - смотри туда.
- Эталонные чарты - живой образец, копируй структуру с них (см. раздел 3).

**Приоритет при конфликте:** `CLAUDE.md` (табу и процесс) > `CONVENTIONS.md`
(устройство) > этот файл (порядок действий) > эталонные чарты. Если этот файл
разошёлся с `CONVENTIONS.md` по существу - прав `CONVENTIONS.md`, а этот файл нужно
поправить.

## 1. Железные правила (не нарушать никогда)

Эти правила старше любых соображений удобства. Нарушение - это дефект, а не стиль.

1. **Никаких длинных тире (em dash, символ U+2014) нигде** - ни в YAML, ни в Markdown, ни в
   коммитах, ни в комментариях, ни в ответах пользователю. Только обычный дефис
   `-`, двоеточие или перестроение фразы.
2. **Никакой кириллицы в исходниках чарта.** Комментарии в `templates/` (Go-шаблоны
   `{{/* ... */}}`) - только на английском. Кириллица допустима исключительно в
   Markdown (`README.md`, `CHANGELOG.md`) и в комментариях файлов значений
   (`values*.yaml`) - это принятая здесь практика. В `values.schema.json`
   человекочитаемые `title`/`description` тоже могут быть на русском (их видит
   пользователь в форме).
3. **Коммиты - на английском, Conventional Commits, scope = имя чарта.**
   `feat(project): ...`, `fix(ingress-gateway): ...`, `chore(namespace): ...`.
   Заголовок и тело - по-английски. Без `Co-Authored-By` и без
   `Generated with Claude Code`.
4. **Все git-операции - через скилл `git-workflow`** (ветки, коммиты, PR/MR,
   релизы). Это единственный источник правды по процессу.
5. **Проверка перед коммитом - обязанность агента** (`helm lint` + `helm template`,
   см. раздел 9). Lefthook на push - бэкстоп, а не замена.
6. **Не оставлять артефакты сборки.** Упакованные чарты (`*.tgz`), вендоринг
   `charts/`, `Chart.lock`, любые бинарники - удалять (они в `.gitignore`).
7. **В `NOTES.txt` не писать shell/`kubectl`/`helm`-команды.** Только текстовая
   сводка: что развёрнуто и перечень созданных ресурсов.
8. **Отвечать пользователю на русском.**

## 2. Что вообще делаем и где источник правды

`console-charts` - это набор Helm-чартов управляемых сервисов платформы. Каждый
чарт - самостоятельная директория верхнего уровня. Чарты публикуются в Harbor и
заказываются через портал Console: `values.schema.json` каждого чарта служит и для
валидации значений, и для генерации формы заказа. Поэтому схема и параметры чарта -
это публичный контракт, а не внутренняя деталь.

Источник правды по устройству чарта - `CONVENTIONS.md`. Прежде чем писать, прочитай
его разделы про нейминг, схему и хелперы, и открой один эталонный чарт целиком.

## 3. Два класса чартов: выбери образец до начала работы

В репозитории два стиля. Определись, к какому относится твой чарт, и копируй именно
его эталон - смешивать нельзя.

### 3.1. Стандартный чарт (5-частный нейминг)

Эталоны: `ingress-gateway`, `egress-gateway`, `waypoint`, `policies`.

- Ресурсы именуются по 5-частной схеме
  `{instanceTag}-{clusterTag}-{kindShort}-{projectTag}-{name}` (детали и реестр
  `kindShort` - в `CONVENTIONS.md`).
- Есть блок `naming` (`instanceTag`/`clusterTag`/`projectTag`), списки сущностей с
  `enabled`, хелперы `<chart>.helpers.*` (`app.fullname`, `tag`, `shortToken`,
  `app.metadata`, ...).
- Это стиль по умолчанию для **нового** сетевого/политик-ресурса.

### 3.2. cpaas-чарт (Alauda/cpaas-нейминг)

Эталоны: `namespace` (`managed-namespace`), `project`.

- Ресурс - объект Alauda/cpaas (`Namespace`, `auth.alauda.io/v1 Project`, ...). Имя
  ресурса берётся напрямую из значения (`namespace.namespaceName`, `project.name`),
  а не собирается по 5-частной схеме.
- labels в cpaas-стиле (`app.cpaas.io/name`, `cpaas.io/*`), нет блока `naming`.
- Есть служебный флаг `global.enable` (консоль включает его при заказе; при `false`
  чарт не создаёт ничего).
- Хелперы `<chart>.helpers.*`: `chart`, `labels`, `metadata` и валидаторы имён.

> Исторически `namespace` использует префикс хелперов `managed-ns.*`, а `console` -
> `console.*`. Для **новых** чартов префикс - строго `<chart>.helpers.*`, где
> `<chart>` совпадает с полем `name` из `Chart.yaml`.

Если сомневаешься - твой ресурс относится к cpaas/Alauda? Бери образцом `project`
или `namespace`. Иначе - `egress-gateway`.

## 4. Процедура: добавить новый чарт

Выполняй по шагам, не пропуская. Каждый шаг - проверяемый.

1. **Заведи ветку** через `git-workflow`. Определи имя чарта = имя директории =
   scope коммитов.
2. **Выбери класс и эталон** (раздел 3). Открой эталон целиком и держи рядом.
3. **`Chart.yaml`**: `apiVersion: v2`, `name` = имя директории, `type: application`,
   `version: "1.0.0"` (SemVer). `appVersion` - только если у чарта есть отдельное
   деплоимое приложение (у чистых CRD-чартов его нет).
4. **`templates/_helpers.tpl`**: заведи хелперы с префиксом `<chart>.helpers.*` -
   `chart`, `labels`, `metadata` (единый блок labels+annotations) и валидаторы
   входных данных. Логика - здесь, тела манифестов - тонкие. Канонические сниппеты -
   раздел 7.
5. **`templates/<Kind>.yaml`**: манифест(ы). Metadata на каждом ресурсе - одним
   вызовом `<chart>.helpers.metadata`. Все инварианты входа проверяй через
   `required`/`fail` (раздел 8).
6. **Файлы значений** (все три, раздел 6):
   - `values.yaml` - минимальные дефолты, лишь бы рендер проходил;
   - `values.minimal.yaml` - рабочий пример;
   - `values.full.yaml` - полный reference с комментариями (на русском).
7. **`values.schema.json`**: draft-07, `additionalProperties: false` на всех
   уровнях, `required`, `propertyOrder`, `title`/`description`. Держи её в синхроне с
   параметрами. Дефолтный `values.yaml` обязан валидироваться схемой.
8. **`templates/NOTES.txt`**: пост-установочная сводка без команд (раздел 1, п. 7).
9. **`README.md`**: для пользователя-человека, не для разработчика чарта. Что это,
   что получится, какие поля заполнить, какие валидации (раздел 6).
10. **`CHANGELOG.md`**: фиксированная шапка + запись `## [1.0.0] - YYYY-MM-DD` с
    категориями (Added/Changed/...).
11. **`.helmignore`**: стандартный набор игнор-паттернов (скопируй с эталона).
12. **Проверка** (раздел 9): `helm lint` + `helm template` по трём наборам значений,
    плюс негативные кейсы (валидации падают, схема ловит опечатки).
13. **Коммит** через `git-workflow`, scope = имя чарта, тело на английском.

## 5. Процедура: изменить существующий чарт

1. Ветка через `git-workflow`.
2. Внеси правку в шаблоны/значения.
3. **Синхронно обнови `values.schema.json`** (новый/удалённый/переименованный ключ)
   и все три файла значений, где он встречается.
4. **Обнови `README.md`**, если изменился пользовательский контракт (поля,
   валидации, поведение).
5. **Bump `version` в `Chart.yaml`** по SemVer: MAJOR - несовместимо с текущим
   `values.yaml` пользователя; MINOR - новая возможность без поломки; PATCH -
   прозрачное исправление.
6. **Добавь запись в `CHANGELOG.md`** сверху, с датой и категорией. BREAKING
   помечай явно и пиши, что поправить в `values.yaml`.
7. Проверка (раздел 9). Коммит через `git-workflow`.

## 6. Чек-лист файлов и пофайловые правила

| Файл                     | Обязателен | Кратко                                                     |
|--------------------------|------------|------------------------------------------------------------|
| `Chart.yaml`             | да         | `name` = имя директории; `version` SemVer                  |
| `values.yaml`            | да         | Минимальные дефолты, лишь бы рендер проходил               |
| `values.minimal.yaml`    | желательно | Рабочий пример (стартовая точка деплоя)                     |
| `values.full.yaml`       | да         | Полный reference всех параметров с комментариями (русский) |
| `values.schema.json`     | да         | draft-07, `additionalProperties: false`, в синхроне с параметрами |
| `templates/_helpers.tpl` | да         | Хелперы `<chart>.helpers.*`: имена, labels, валидации      |
| `templates/*.yaml`       | да         | Тонкие манифесты; metadata через хелпер                    |
| `templates/NOTES.txt`    | да         | Сводка без команд                                          |
| `README.md`              | да         | Для пользователя: что, зачем, какие поля, какие валидации  |
| `CHANGELOG.md`           | да         | История версий по SemVer                                   |
| `.helmignore`            | да         | Стандартные игнор-паттерны                                 |

Роли трёх файлов значений (частая ошибка - путать их):

- `values.yaml` - не «осмысленный деплой», а минимум для валидного рендера. У
  cpaas-чартов тут обычно `global.enable: false`, и чарт рендерит пусто - это норма.
  Файл обязан лежать в корне и не может отсутствовать (Helm ждёт его, в т.ч. при
  подключении сабчартом).
- `values.minimal.yaml` - минимальный **рабочий** пример: включает чарт и даёт один
  создаваемый ресурс.
- `values.full.yaml` - документация значений: все параметры с русскими
  комментариями. Не подменяй им `values.yaml`.

README пиши простым языком и по делу (что это, что получится, как заказать, какие
поля). Тяжёлый reference (все параметры, логика рендера) живёт в `values.full.yaml`
и `values.schema.json`, а не в README. Обязательно перечисли раздел «Валидации» -
зеркало всех `fail`/`required` из шаблонов.

Подробные каркасы `Chart.yaml`, `values.schema.json`, `CHANGELOG.md` и структуру
README смотри в `CONVENTIONS.md` - не переписывай их по памяти.

## 7. Шаблонизация: канонические сниппеты

Префикс всех хелперов - `<chart>.helpers.*` (чтобы не конфликтовать при
использовании чарта сабчартом). Комментарии к `define` - короткий header на
английском. Вся общая логика - в `_helpers.tpl`; тела манифестов тонкие.

**Единый блок metadata.** labels и annotations на каждом ресурсе выводятся ОДНИМ
вызовом, а не двумя. Хелпер рендерит полный `labels:` и, только когда есть что
выводить, блок `annotations:` (сюда автоматически попадают `generic.*`). Так
`generic.labels`/`generic.annotations` нельзя забыть на новом манифесте, и не
возникает двух ключей `annotations:` в одном metadata. Вызов ставится сразу под
`metadata.name`:

```gotemplate
metadata:
  name: {{ $name }}
  {{- include "<chart>.helpers.metadata" (dict "context" . "labels" (dict "cpaas.io/project" $name) "annotations" $ann) | nindent 2 }}
```

Ресурс-специфичные labels/annotations передаются параметрами `labels`/`annotations`
в тот же вызов - отдельного блока `annotations:` в манифесте быть не должно.

**Резолв флага `enabled`.** Для списков сущностей используй хелпер, уважающий явный
`false`. Не пиши `.enabled | default true` - это превращает явный `false` в `true`:

```gotemplate
{{- if hasKey $entity "enabled" -}}
{{- ternary "true" "" (eq (toString $entity.enabled | lower) "true") -}}
{{- else -}}true{{- end -}}
```

Полный набор стандартных хелперов (`app.chart`, `app.name`, `tag`, `shortToken`,
`app.fullname`, `app.selectorLabels`, `app.labels`, `app.metadata`, `app.enabled`,
`tplvalues.render`) описан в `CONVENTIONS.md`; готовый код - в `egress-gateway`
(стандартный класс) и `project` (cpaas-класс).

## 8. Валидация значений: defense in depth

Невалидный ввод обязан ронять рендер с понятным сообщением, а не молча создавать
кривой манифест. Две линии обороны, обе обязательны:

1. **`values.schema.json`** - первая линия. Ловит опечатки в ключах
   (`additionalProperties: false`), обязательность (`required`), формат/длину
   (`pattern`, `maxLength`, `minItems`, `enum`). Форма портала не даст отправить
   заведомо невалидное.
2. **`required`/`fail` в шаблонах** - бэкстоп на инварианты, которые схема не
   выражает, и на случай рендера в обход портала:

```gotemplate
{{- $name := required "project.name is required" .Values.project.name | toString | lower -}}
{{- if not (regexMatch "^[a-z0-9]([-a-z0-9]*[a-z0-9])?$" $name) -}}
{{- fail (printf "project.name must be DNS-like lowercase (RFC 1123), got %q" $name) -}}
{{- end -}}
```

`required` - для обязательных полей, `fail` - для нарушенных инвариантов (длина,
формат, неизвестный enum/`kindShort`, пустой список). Каждое условие падения
перечисли в разделе «Валидации» README - README и шаблоны должны совпадать.

## 9. Обязательная проверка перед коммитом

Локально, до коммита (lefthook на push - бэкстоп):

```sh
helm lint <chart> -f <chart>/values.minimal.yaml
helm template release <chart>                                # дефолтный values.yaml
helm template release <chart> -f <chart>/values.minimal.yaml
helm template release <chart> -f <chart>/values.full.yaml
```

Рендер обязан проходить чисто и валидироваться по `values.schema.json`.
Дополнительно прогони негативные кейсы: подсунь невалидное `name`, пустой список,
лишний ключ - убедись, что схема и/или `fail` роняют рендер с внятным сообщением.

Прогнать весь репозиторий разом: `sh scripts/helm-check.sh` (или
`lefthook run pre-push`). Ожидаемый финал - `all charts valid`.

## 10. Типичные ошибки агента (антипаттерны)

Перед коммитом пройдись по списку - это то, что агенты ломают чаще всего:

- **Длинное тире (em dash, U+2014)** просочилось в комментарий/README/коммит. Заменить на `-`.
- **Кириллица в `templates/`** (комментарий или, хуже, в значении). В шаблонах -
  только латиница.
- **`| default true`** для `enabled` - явный `false` теряется. Использовать хелпер
  `enabled` (раздел 7).
- **Два блока `annotations:`** в одном metadata (структурные + `generic.*`) - это
  невалидный YAML. Всё через один вызов `metadata`.
- **Схема разошлась с параметрами** - добавил ключ в шаблон, забыл в
  `values.schema.json` (и наоборот). Правь синхронно, в одном изменении.
- **Забыт bump `version` и запись в `CHANGELOG.md`** при правке чарта.
- **`values.yaml` не рендерится** или требует ручного ввода. Дефолты обязаны давать
  валидный рендер (или явно падать через `required`/`fail`), а не молча ломаться.
- **Команды в `NOTES.txt`** (`kubectl`/`helm`). Только текстовая сводка.
- **Оставлен артефакт** (`*.tgz`, бинарник) или вложенный `.git` внутри папки чарта.
  В монорепо чарт - обычная папка, без своего `.git`.
- **Скопирован не тот класс** (5-частный нейминг в cpaas-чарт или наоборот). Сверься
  с разделом 3.
- **Префикс хелперов не `<chart>.helpers.*`.** Исторические `managed-ns.*`/`console.*`
  не копируй в новый чарт.

## 11. Definition of Done

Чарт готов, когда:

- [ ] Все обязательные файлы на месте (раздел 6), `.helmignore` есть.
- [ ] `Chart.yaml`: `name` = директория, `version` по SemVer выставлен/поднят.
- [ ] `values.schema.json` в синхроне со всеми тремя файлами значений.
- [ ] Все инварианты покрыты `required`/`fail` и перечислены в README «Валидации».
- [ ] metadata на каждом ресурсе - через единый хелпер; `generic.*` доезжают.
- [ ] `helm lint` + `helm template` (default/minimal/full) проходят; негативные
      кейсы падают внятно; `scripts/helm-check.sh` даёт `all charts valid`.
- [ ] `CHANGELOG.md` содержит запись текущей версии.
- [ ] Нет длинного тире, кириллицы в исходниках, команд в `NOTES.txt`, артефактов
      сборки и вложенного `.git`.
- [ ] Коммит через `git-workflow`, scope = имя чарта, тело на английском.
